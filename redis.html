Redis

In-memory data structure store — database, cache, message broker, and streaming engine.

🔍

Core Concepts

⚡ In-Memory
All data stored in RAM for sub-millisecond latency. Optionally persists to disk via RDB snapshots or AOF logs.
🧩 Data Structures
Native support for Strings, Hashes, Lists, Sets, Sorted Sets, Streams, Bitmaps, HyperLogLog, and Geospatial indexes.
📡 Pub/Sub
Built-in publish/subscribe messaging. Clients subscribe to channels and receive real-time messages. Fire-and-forget semantics.
🔄 Replication
Leader-follower replication with async sync. Redis Sentinel for HA. Redis Cluster for horizontal sharding across nodes.

Common Patterns

🗄️ Cache-Aside
Check Redis first. On miss, query DB, then write to Redis with TTL. 
GET cache:user:1
// miss → query DB → SET cache:user:1 "data" EX 3600
🔒 Distributed Lock
Use SET NX EX for mutual exclusion across processes. 
SET lock:resource "owner" NX EX 30
// Do work...
DEL lock:resource
⏱️ Rate Limiting
Use INCR with EXPIRE for sliding window rate limiting. 
INCR ratelimit:user:1
EXPIRE ratelimit:user:1 60
// Check if count > threshold
📊 Leaderboard
Sorted sets are perfect for leaderboards with O(log N) operations. 
ZADD leaderboard 1500 "player1"
ZREVRANGE leaderboard 0 9 WITHSCORES

Persistence Options

MethodHow It WorksTrade-off
RDBPoint-in-time snapshots at intervalsFast restart, possible data loss between snapshots
AOFLogs every write commandMore durable, larger files, slower restart
RDB + AOFCombined — AOF for durability, RDB for fast recoveryRecommended for production
No persistencePure cache, data lost on restartBest performance, no durability
